# ADR-001 - Cache-First con Source of Truth Locale

## Context
Il progetto myFitnessMeals richiede UX veloce per ricerca/logging alimenti, supporto offline robusto (RNF-005), fallback Open Food Facts su cache miss e persistenza override nutrienti utente (RF-010, RF-011). L'app deve inoltre mantenere dashboard e storico affidabili anche con rete instabile.

## Decision
Adottare un'architettura cache-first con database locale Room come sorgente autorevole per il dominio applicativo. Le chiamate remote (OFF, Garmin) aggiornano la cache tramite repository/adapters, senza bypassare la source of truth locale.

Policy chiave:
- Letture: locale prima, remoto solo su cache miss o refresh programmato.
- Scritture utente: sempre su locale in transazione.
- Priorita nutrienti: Override > Cache > OFF.
- Sync fitness asincrona via WorkManager con idempotenza per data/provider.

## Alternatives considered
- Solo rete con cache in-memory:
  - Pro: implementazione iniziale piu rapida.
  - Contro: offline scarso, latenza alta, resilienza bassa.
- Backend centrale obbligatorio per orchestrare catalogo e sync:
  - Pro: controllo completo su dati e rate limiting.
  - Contro: costo/complessita elevata, non necessaria per MVP.
- Remote-first con persistenza opportunistica:
  - Pro: dati piu freschi in teoria.
  - Contro: UX degradata in rete debole, rischio non conformita AC offline.

## Consequences
- Positive:
  - Prestazioni locali prevedibili (ricerca/salvataggio p95).
  - Esperienza offline consistente.
  - Dominio testabile e stabile rispetto a variazioni API esterne.
- Negative:
  - Maggiore complessita su migrazioni DB e politiche di invalidazione cache.
  - Necessita di meccanismi deduplica/normalizzazione per dati OFF eterogenei.

## Notes
- Decisione coerente con scope MVP e con possibile estensione a provider multipli (Fitbit/Google Fit) tramite adapter aggiuntivi.
- Riesame ADR previsto alla fine della beta, in base a metriche reali: cache-hit ratio, sync-success rate, incidenti di data quality.
